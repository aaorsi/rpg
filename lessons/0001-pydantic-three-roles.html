Pydantic's three roles in this codebase

Lesson 0001 · tied to Mission: Pydantic in the RPG policy orchestrator

In services/policy_orchestrator, Pydantic is not decorative typing — it is the contract layer between Unity, the sidecar, and Ollama. Every dialogue turn passes through it three times.

The pipeline (one dialogue turn)

Unity POST /v1/dialogue/turn (camelCase JSON) ↓ FastAPI → DialogueTurnRequest.model_validate() ← Role 1: guard inputs ↓ Ollama returns messy text (fences, synonyms) ↓ dialogue_parsing.parse_dialogue_output() ← tolerant pre-parse ↓ LlmDialogueOutput.model_validate() ← Role 2: enforce LLM shape ↓ PolicyEnvelope(ok=True, dialogue=…) ← Role 3: uniform response ↓ Unity reads camelCase JSON

Role 1 — Guard what Unity sends in

StrictCamelModel on request types like DialogueTurnRequest.
Python field names are snake_case; wire JSON is camelCase via alias_generator=to_camel.
extra="forbid" means a typo like bogusField fails immediately — see tests/test_models.py.

Role 2 — Enforce what the LLM must emit

LlmDialogueOutput in app/models.py is the single source of truth for dialogue turn JSON shape.
The LLM is unreliable, so dialogue_parsing.py coerces messy output into a clean dict first, then model_validate enforces types and required fields (say is mandatory).
Same pattern for narrative: SessionNarrativeCanon.model_validate(canon) in orchestrator.py.

Role 3 — Generate schemas Unity can trust

schema_export.py calls LlmDialogueOutput.model_json_schema(by_alias=True) and writes to
Assets/StreamingAssets/Dialogue/schema/dialogue_turn_result.schema.json.
Regenerate with python scripts/generate_schemas.py.
test_schema_sync.py fails if committed schemas drift from models — this is your safety net when you add a field.

Why two base classes?

CamelModel (extra="ignore") for LLM outputs — models invent extra keys constantly; drop them. StrictCamelModel (extra="forbid") for Unity requests — your code sent the field; typos should fail loud.
